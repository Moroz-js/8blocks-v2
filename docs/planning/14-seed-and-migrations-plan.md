# Seed and Migrations Plan

## 1. Purpose

Описать стратегию миграций БД и отдельный seed-скрипт для локальной разработки и первичного наполнения: состав моковых данных, порядок выполнения, безопасность (не перезаписывать прод).

## 2. Scope

- Как создаются и применяются миграции (Payload/PostgreSQL).
- Отдельный seed-скрипт: тестовые статьи, категории, теги, SEO, связи; при необходимости тестовые медиа, заявки, подписки.
- Запуск: когда и как; идемпотентность или одноразовый запуск.

## 3. Inputs from requirements

- Структура данных поддерживается через миграции.
- Для локальной разработки и первичного наполнения — отдельный seed-скрипт.
- Моковые данные: тестовые статьи, категории, теги, SEO-поля, связанные сущности.

## 4. Migrations strategy

### 4.1 Инструмент

- Payload 3.x с PostgreSQL adapter обычно генерирует миграции при изменении коллекций (команда generate: migrate или аналог). Либо миграции вручную (Drizzle/Prisma), если схема ведётся отдельно. В плане предполагаем использование миграций Payload при наличии.
- Хранить миграции в репозитории (папка migrations/ или payload/migrations/). Не менять схему напрямую в БД без миграции.

### 4.2 Порядок применения

- При деплое и при локальной установке: сначала `migrate` (применить все pending миграции), затем при необходимости `seed`. Seed не должен зависеть от конкретного состояния данных продовой БД — только от схемы.

### 4.3 Naming и версионирование

- Имена файлов миграций с timestamp или порядковым номером; описание в имени или в комментарии (например «add noindex to articles»).

## 5. Seed strategy

### 5.1 Назначение

- Локальная разработка: быстро получить работающий блог с категориями, тегами и статьями для проверки списков, фильтров, страницы статьи, ToC, SEO.
- Демо/тестовое окружение: предзаполненный контент.
- Seed не предназначен для production без явного решения (например «первичное наполнение один раз» с защитой от повторного запуска).

### 5.2 Состав моковых данных

- **Categories**: 3–5 категорий с title, slug, description, seoTitle, seoDescription. Slug уникальные (например tokenomics, company, news).
- **Tags**: 5–10 тегов с name и slug.
- **Articles**: 5–15 статей в статусе published и 1–2 в draft. Поля: title, slug (уникальные), excerpt, richText (с заголовками h2/h3, параграфами, цитатой, ссылкой, изображением если есть тестовое медиа), category (связь с созданными категориями), tags (несколько), publishedAt, seoTitle, seoDescription, noindex (часть статей false, одна — true для проверки). Одна статья с обложкой при наличии тестового медиа.
- **Media** (опционально): 2–3 тестовых изображения в public/uploads для обложек и rich text; записи в коллекции Media.
- **Leads** (опционально): 2–3 тестовые заявки для отображения в админке.
- **NewsletterSubscriptions** (опционально): 2–3 тестовых email.

### 5.3 Порядок создания в seed

1. Categories (нет зависимостей).
2. Tags (нет зависимостей).
3. Media (если нужны файлы — сначала скопировать/сгенерировать в uploads, затем создать записи Media).
4. Articles (зависимости: category, tags, опционально cover → Media).
5. Leads, NewsletterSubscriptions (опционально).

### 5.4 Идемпотентность и безопасность

- **Вариант A (рекомендуемый для локальной разработки)**: seed проверяет переменную NODE_ENV или SEED_ALLOWED; если не development или флаг не установлен — не выполнять или выйти с предупреждением. При запуске в dev — очищать коллекции Articles, Tags, Categories (и при необходимости Media, Leads, Subscriptions) и создавать заново. Либо использовать уникальные slug/имена и проверять существование; при существовании — пропускать создание.
- **Вариант B**: seed только добавляет данные без удаления; проверка по slug — если категория/тег/статья с таким slug есть, пропуск. Это позволит запускать seed несколько раз без дубликатов, но не перезапишет существующие данные.
- Для production: не запускать seed по умолчанию; отдельная команда или флаг SEED_PRODUCTION (крайне осторожно, только для первичного наполнения).

### 5.5 Реализация

- Скрипт: scripts/seed.ts (или .js). Запуск: `npx ts-node scripts/seed.ts` или `tsx scripts/seed.ts`; в package.json: `"seed": "tsx scripts/seed.ts"` (или с проверкой env).
- Подключение к БД: тот же DATABASE_URI, что и у приложения. Использовать Payload API (getPayload()) или прямой доступ к БД через адаптер; предпочтительно через Payload local API для согласованности валидации и хуков.
- Rich text в статьях: формировать структуру блоков в формате Payload (lexical/slate) — минимальный набор: заголовки, параграфы, блок с upload (если есть тестовое медиа). Либо использовать фиксированные JSON-файлы с примером контента и подставлять в seed.

## 6. Dependencies

- Миграции должны быть применены до seed (схема актуальна).
- Коллекции и поля соответствуют 08-cms-plan и 09-data-models-and-db-plan.
- Seed выполняется после Phase 2 (Payload + БД готовы).

## 7. Risks / open questions

- Очистка коллекций в seed может удалить ручные данные в dev; предупреждать в README. Либо не очищать, а только добавлять при отсутствии (по slug).
- Формат rich text Payload может меняться; seed должен использовать актуальный формат, иначе статьи не отобразятся корректно.

## 8. Deliverables

- Документированная команда применения миграций (например `pnpm payload migrate` или аналог).
- Скрипт scripts/seed.ts с созданием категорий, тегов, статей (и опционально медиа, заявок, подписок).
- README: как запустить миграции и seed локально; предупреждение о production.
- .env.example: без изменений; seed использует DATABASE_URI.

## 9. Acceptance criteria

- После применения миграций схема БД соответствует коллекциям Payload.
- Запуск seed в development создаёт тестовые категории, теги и статьи; блог отображает данные; одна статья с noindex; в админке видны draft и published.
- Seed не выполняется в production без явного разрешения (флаг или отсутствие в деплой-скриптах).
- Документация обновлена.
