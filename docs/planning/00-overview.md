# Planning Overview — 8Blocks

## 1. Purpose

Этот документ задаёт общий контекст проекта и служит точкой входа в систему planning-документов. По нему можно понять цели, ограничения, стек и стратегию реализации, а также найти нужный детальный план по области.

## 2. Описание проекта

**8Blocks** — контентная платформа: корпоративный сайт + блог. Целевой результат:

- Публичный сайт с главной, страницами услуг (все услуги, Strategic Consulting, Tokenomics, Audit), блогом (архив, категории, статьи), privacy policy и 404.
- Админ-панель на Payload CMS (встроена в Next.js): статьи, категории, теги, медиа, заявки, подписки на рассылку.
- Формы обратной связи и подписки с сохранением в БД и отправкой email (пользователь + администратор).
- SEO: метаданные, canonical, OG/Twitter, structured data (Organization, BlogPosting), динамические sitemap.xml и robots.txt.
- Локальное хранение медиа в `/public/uploads`, серверная валидация загрузок.

Мультиязычность не требуется. Язык интерфейса и контента — по решению (в ТЗ допускаются служебные названия секций на английском при русском контенте).

## 4. Ограничения (из ТЗ)

| Область | Ограничение |
|--------|--------------|
| Стек | Next.js, TypeScript, SCSS Modules + global.scss, Mantine, Framer Motion, Lenis, clsx, Swiper/Embla только при необходимости, Payload CMS, PostgreSQL |
| Mantine | Только перечень компонентов из ТЗ (Button, TextInput, Textarea, Select, Menu, Modal, Accordion, Tabs, Checkbox, Radio, Switch, Drawer, Notification); не задаёт общий визуальный стиль |
| UI | Основной UI — кастом через SCSS Modules и проектные компоненты |
| CMS | Payload встроен в Next.js, без отдельного backend (не Strapi-подход) |
| Медиа | Только локальное хранение в `/public/uploads` |
| Email | SMTP, конфиг через env |
| Админ | Логин/пароль из env; роуты по согласованной структуре |
| Данные | PostgreSQL, структура через миграции, seed для моков |

## 5. Ключевые архитектурные принципы

- **FSD-подобная структура**: разграничение app, pages, widgets, features, entities, shared; чёткие границы слоёв и import rules.
- **CMS как часть приложения**: Payload-конфиги и коллекции живут в проекте; публичный сайт и админка — один Next.js app.
- **Страницы услуг — не шаблон**: каждая услуга — отдельный лендинг с собственной структурой секций и ритмом.
- **Статья как first-class**: slug, excerpt, rich text, обложка, категория, теги, статус, SEO, noindex, ToC из контента, просмотры через API.
- **SEO и контент**: единая стратегия метаданных, canonical, sitemap/robots; контент — явное разделение «в коде» vs «в CMS».

## 6. Общая стратегия реализации

1. **Phase 0**: инициализация проекта, зависимости, базовая конфигурация, design tokens.
2. **Phase 1**: каркас архитектуры (FSD, роутинг, layout, общие компоненты).
3. **Phase 2**: Payload + БД (коллекции, миграции, seed) — фундамент для контента и форм.
4. **Phase 3**: главная страница (все секции, тексты, якоря, footer/форма).
5. **Phase 4**: страницы услуг (все услуги + три лендинга с уникальными блоками и FAQ).
6. **Phase 5**: блог (архив, категории, статья: rich text, ToC, просмотры, метаданные).
7. **Phase 6**: формы (contact, newsletter), персистенция, email (SMTP, шаблоны).
8. **Phase 7**: SEO (metadata builders, sitemap, robots, structured data), GTM.
9. **Phase 8**: полировка, 404, privacy policy, QA, DoD.

Подробный порядок и критерии — в `16-delivery-phases.md`.
