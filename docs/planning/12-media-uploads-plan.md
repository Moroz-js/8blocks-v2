# Media and Uploads Plan

## 1. Purpose

Описать политику хранения, валидации и доступа к загружаемым файлам: /public/uploads, обложки статей, изображения в rich text, публичный URL, безопасность и обработка ошибок.

## 2. Scope

- Директория хранения: /public/uploads.
- Валидация: тип файла, размер, безопасное имя.
- Использование: обложка статьи (Payload Media → Article.cover), изображения в rich text (upload в блоке).
- Публичный URL и отдача файлов.
- Ошибки загрузки и отказоустойчивость.

## 3. Inputs from requirements

- Все загружаемые файлы хранятся локально в /public/uploads.
- Серверная загрузка с валидацией: тип файла, размер, безопасное имя.
- Файлы доступны по публичному URL в рамках проекта.
- Загрузка используется для обложек статей и изображений внутри rich text.

## 4. Storage policy

- **Путь на диске**: public/uploads. Payload upload adapter настроен на эту директорию (local storage).
- **Структура поддиректорий**: по желанию — плоская (все файлы в uploads/) или по типу/дате (uploads/images/2025/03/). Для простоты — плоская с уникальным именем файла (см. naming).
- **Git**: public/uploads/* в .gitignore; в репо не коммитить загруженные файлы; при необходимости .gitkeep в uploads/ для сохранения папки.

## 5. Naming policy

- **Безопасное имя**: убрать небезопасные символы (путь, спецсимволы); оставить буквы, цифры, дефис, подчёркивание; расширение — по MIME. Дубликаты: добавить суффикс (timestamp или uuid) к имени до расширения, чтобы не перезаписывать.
- **Пример**: исходное `image 1 (2).png` → `image-1-2-1734567890.png` или `uuid.png`.
- Реализация в Payload: использовать встроенные опции sanitize filename или кастомный hook beforeChange для Media.

## 6. Validation policy

- **Тип файла**: разрешить только image/* (jpeg, png, gif, webp) для контента статей и обложек. При необходимости в будущем — PDF для документов; на первом этапе только изображения.
- **Размер**: лимит на файл (например 5 MB или 10 MB) — задать в Payload config и при необходимости в next.config (bodyParser limit). Конкретное значение — по решению (указать в конфиге).
- **Расширение и MIME**: проверять соответствие (белый список MIME и расширений); отклонять исполняемые и подозрительные типы.
- Серверная валидация обязательна; клиентская — опционально (предупреждение до загрузки).

## 7. Article cover flow

- В статье (Payload) поле cover — relation к Media. Редактор в админке выбирает загруженное изображение или загружает новое. На фронте при рендере статьи запрашивается URL медиа: например /uploads/filename или через Payload API URL поля. Отображение: img с src, alt из Media.alt.

## 8. Rich text image flow

- В rich text блок типа upload (image) ссылается на Media. При рендере rich text на фронте для каждого такого блока подставляется img с src=/uploads/... и alt из медиа (или из блока). Responsive: при необходимости srcset (Payload или фронт генерирует несколько размеров — опционально на первом этапе).

## 9. Public URL strategy

- **Вариант A**: статика Next.js — файлы в public/uploads доступны по URL /uploads/filename. Next.js отдаёт их как статические файлы.
- **Вариант B**: отдельный route (например /api/uploads/...) с проверкой прав — не требуется по ТЗ (публичный доступ).
- Итог: URL вида `https://site.com/uploads/filename` или относительный `/uploads/filename`. Сборка полного URL для OG image: SITE_URL + /uploads/filename.

## 10. Security considerations

- Не выполнять загруженные файлы; только отдавать как статику с корректным Content-Type.
- Валидация типа и размера на приёме; запрет на загрузку .php, .exe и т.п. под видом изображения (проверка по содержимому magic bytes — опционально, рекомендуется).
- Ограничение доступа: по ТЗ uploads публичны; при необходимости в будущем — закрыть служебные пути или отдавать через API с проверкой реферера (не в текущем scope).

## 11. Failure states

- **Загрузка не удалась**: валидация не пройдена — вернуть ошибку в админке с указанием причины (тип/размер/имя). Ошибка диска — логировать, вернуть общее сообщение.
- **Файл удалён с диска, но запись в Media осталась**: при отображении статьи/rich text проверять наличие файла или показывать placeholder/ломаную картинку; в админке при выборе медиа показывать превью только если файл доступен. Опционально: cron или хук для очистки orphan Media.
- **Нет прав на запись в public/uploads**: при старте приложения или первой загрузке проверять доступность директории; в документации указать необходимость прав на запись.

## 12. Dependencies

- Payload config (08): коллекция Media с destination public/uploads; хуки валидации при необходимости.
- Статья и rich text renderer (07, 08): использование URL медиа при рендере.

## 13. Risks / open questions

- Большой объём файлов: одна плоская папка может замедлить FS; при росте рассмотреть поддиректории по дате или по id.
- CDN: в ТЗ только локальное хранение; при переходе на CDN URL-стратегия изменится (конфигурируемый base URL для медиа).

## 14. Deliverables

- Настройка Payload upload: destination public/uploads, лимит размера, допустимые MIME.
- Санитизация имени файла (встроенная или hook).
- Документация: допустимые форматы и размер; как вставить изображение в статью и задать обложку.
- .gitignore: public/uploads/* (с возможным исключением .gitkeep).

## 15. Acceptance criteria

- Загрузка через админку сохраняет файл в public/uploads с безопасным именем.
- Валидация типа и размера срабатывает; при нарушении загрузка отклоняется с понятной ошибкой.
- Обложка статьи и изображения в rich text отображаются по публичному URL /uploads/...
- Удаление медиа в админке (если реализовано) не ломает отображение статей без обложки; обработанные кейсы «файл отсутствует» по решению (placeholder или скрытие).
