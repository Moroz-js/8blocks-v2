# Blog and Article System Plan

## 1. Purpose

Глубокая декомпозиция раздела блога: архив, страница категории, страница статьи; rich text, ToC, slug, статусы, просмотры, метаданные, пагинация, фильтрация. Связь CMS и frontend-рендеринга.

## 2. Scope

- Маршруты: /blog, /blog/c/[category], /blog/[slug].
- Архив: список статей, карточки, пагинация, базовая фильтрация, переход в категорию и в статью.
- Категория: название, список статей категории, пагинация, SEO.
- Статья: breadcrumbs, заголовок, служебная информация, обложка, rich text, ToC, метаданные, noindex, просмотры, время чтения.
- Правила slug, draft/published, чтение из Payload.

## 3. Inputs from requirements

- Блог: архив со списком статей, карточки, пагинация, фильтрация, переход в категорию и статью.
- Категория: название, список статей, пагинация, SEO категории.
- Статья: breadcrumbs, заголовок, служебная информация, обложка, rich text, ToC (автоматически из заголовков), SEO.
- Структура статьи: заголовок, slug, excerpt, rich text, обложка, категория, теги, статус, дата публикации, SEO-поля, noindex, даты создания/обновления.
- ToC: парсинг заголовков из HTML, id у заголовков, якорные ссылки.
- Просмотры: учёт через API при открытии статьи; не чаще одного раза за сессию.
- Время чтения: не хранится в CMS, вычисляется по контенту. Автор фиксированный: 8Blocks.
- Slug уникален; статусы draft / published.

## 4. Routing (напоминание)

- `/blog` — архив.
- `/blog/c/[category]` — страница категории (slug категории).
- `/blog/[slug]` — страница статьи (slug статьи). Резолв: только статьи; при отсутствии — 404.

Важно: slug статей и slug категорий не пересекаются (разные коллекции и разные пути).

## 5. Архив блога (/blog)

### 5.1 Данные

- Список статей со статусом published; сортировка по дате публикации (новые первые).
- Пагинация: limit + offset или cursor; параметр страницы ?page=2 или сегмент пути (по решению).
- Фильтрация: по категории (ссылки на /blog/c/[category]), по тегу (query или отдельная страница — в ТЗ «базовая фильтрация»; минимум — фильтр по категории через переход на страницу категории).
- Для карточек нужны: title, excerpt, slug, category (название + slug), обложка (URL), дата публикации; опционально теги.

### 5.2 UI

- Заголовок секции «Блог» или аналог.
- Сетка карточек статей (ArticleCard): обложка, заголовок, excerpt, дата, категория, ссылка на /blog/[slug].
- Пагинация: кнопки или номера страниц; доступ к ?page=.
- Боковая панель или блок «Категории»: ссылки на /blog/c/[category] для каждой категории (данные из Payload).
- Базовая фильтрация: выбор категории ведёт на страницу категории; по тегу — при наличии требований можно фильтровать список на той же странице по query.

### 5.3 Deliverables

- app/blog/page.tsx; данные через getPayload() или API; пагинация и фильтры.
- Виджет BlogArchive (сетка + пагинация + список категорий).
- ArticleCard: переиспользуемый компонент (архив и страница категории).

### 5.4 Acceptance

- Отображаются только published статьи.
- Карточки ведут на статью; ссылки на категории ведут на /blog/c/[category].
- Пагинация работает; SEO метаданные страницы архива заданы (11).

## 6. Страница категории (/blog/c/[category])

### 6.1 Данные

- По slug категории получить категорию; если не найдена — 404.
- Список статей этой категории (relation или фильтр по categoryId); только published; сортировка по дате; пагинация.
- SEO: title, description из полей категории (если есть в CMS) или сгенерированные.

### 6.2 UI

- Breadcrumbs: Главная → Блог → [Название категории].
- Заголовок — название категории.
- Та же сетка карточек статей + пагинация.
- Состав ArticleCard тот же.

### 6.3 Deliverables

- app/blog/c/[category]/page.tsx; resolve category by slug; fetch articles; generateMetadata.
- Виджет BlogCategoryPage или переиспользование сетки архива с другим заголовком и breadcrumbs.

### 6.4 Acceptance

- 404 при неизвестном slug категории.
- SEO категории (11); пагинация.

## 7. Страница статьи (/blog/[slug])

### 7.1 Данные

- Статья по slug; только published (в админке видны и draft; на фронте — только published). Не найдена → 404.
- Поля: title, slug, excerpt, richTextContent, cover (media), category, tags, publishedAt, author (фиксированный 8Blocks), seoTitle, seoDescription, noindex, createdAt, updatedAt.
- Время чтения: вычислять на фронте или в shared/lib по длине текста (слов или символов).
- Просмотры: при первом открытии статьи в сессии — вызов API (например POST /api/article/[slug]/view); счётчик хранится отдельно (в БД или кэше); не чаще одного раза за сессию (cookie или sessionStorage).

### 7.2 Структура страницы

1. **Breadcrumbs** — Главная → Блог → [Категория] → [Заголовок статьи].
2. **Заголовок** — h1.
3. **Служебная информация** — дата публикации, автор (8Blocks), время чтения (вычисленное).
4. **Обложка** — при наличии; responsive image.
5. **Rich text контент** — рендер из Payload rich text (заголовки, параграфы, цитаты, ссылки, изображения с alt). Заголовкам в рендере добавлять id (slug от заголовка или автоинкремент) для якорей ToC.
6. **ToC** — боковая панель или верхний блок; генерируется парсингом HTML/структуры контента после рендера; пункты ToC — ссылки #id на заголовки внутри статьи.
7. **Метаданные** — generateMetadata: title (seoTitle || title), description (seoDescription || excerpt), canonical, openGraph, twitter, article (publishedTime и т.д.), noindex при флаге; structured data BlogPosting (11).

### 7.3 ToC: реализация

- **Вариант A**: парсить не HTML, а структуру rich text из Payload (блоки типа heading с level) и строить список пунктов с id; при рендере заголовкам задавать id.
- **Вариант B**: после рендера rich text в DOM парсить уже отрендеренный HTML (headings h2, h3), добавлять id, строить массив { id, text, level } и рендерить ToC из него.
- Рекомендация: использовать структуру Payload rich text (блоки) при рендере — для каждого заголовка генерировать id (например slug от текста или `h-1`, `h-2`); отдельная функция buildTocFromBlocks() в shared/lib или features; компонент ArticleToc получает массив и рендерит список ссылок.
- Якорные ссылки: при клике скролл к элементу с id; совместимость с Lenis.

### 7.4 Rich text renderer

- Payload отдаёт rich text в структурированном виде (блоки/ноды). На фронте — кастомный рендерер (shared/render): маппинг типов нод на React-компоненты (Heading, Paragraph, Blockquote, Link, Image). Для изображений — URL из media (uploads); поддержка alt. Стили из design system.

### 7.5 Просмотры

- API route: POST /api/articles/[slug]/view (или в рамках Payload API кастомный endpoint). Логика: увеличить счётчик просмотров для статьи; проверка «один раз за сессию» — по cookie (например article_view_[slug]=1) или по sessionStorage. Счётчик может храниться в отдельной таблице/документе (ArticleView или поле в Article — в ТЗ сказано «учитывается отдельно через API», значит предпочтительно отдельное хранилище или отдельное поле, не «постоянное поле в CMS» в смысле ручного ввода; допустимо поле views в Article, обновляемое API).
- ТЗ: «Количество просмотров не хранится как постоянное поле в CMS и учитывается отдельно через API» — интерпретация: в админке поле просмотров может быть read-only и заполняться из внешнего учёта (API), либо счётчик в отдельной сущности; при открытии статьи вызывается API, который увеличивает счётчик. Реализация: поле views в Article, обновляемое только через API при просмотре — допустимо; «не хранится» может означать «не редактируется вручную». Либо отдельная коллекция/таблица ViewCount. Для плана оставляем: API увеличивает счётчик (хранилище — на выбор: поле в Article или отдельная таблица); в админке просмотры можно показывать (read-only).

### 7.6 Deliverables

- app/blog/[slug]/page.tsx; получение статьи по slug; generateMetadata с учётом noindex; structured data BlogPosting.
- Виджеты: Breadcrumbs, ArticleHeader, ArticleCover, ArticleRichContent (с поддержкой id у заголовков), ArticleToc.
- shared/render (или lib): rich text → React; buildTocFromBlocks (или из HTML).
- entities или lib: getReadingTimeMinutes(content).
- API route: POST /api/articles/[slug]/view (или аналог); middleware или проверка сессии/cookie.
- Страница отдаёт 404 для draft или несуществующего slug.

### 7.7 Acceptance

- Статья отображается с breadcrumbs, заголовком, датой, автором, временем чтения, обложкой, rich text, ToC с якорями.
- ToC строится автоматически из заголовков контента; заголовки имеют id.
- Просмотр учитывается один раз за сессию через API.
- noindex применяется в метаданных при установленном флаге.
- Draft не отображается на фронте; 404 при отсутствии статьи.

## 8. Зависимости

- **CMS**: коллекции Articles, Categories, Tags, Media (08, 09).
- **SEO**: 11 (metadata, BlogPosting, canonical).
- **Uploads**: обложка и изображения в rich text — 12.
- **Контент**: автор 8Blocks — константа; время чтения — вычисление (09, 15).

## 9. Риски и открытые вопросы

- Конфликт slug категории и статьи: исключён разными путями (/blog/c/ и /blog/).
- Производительность: большие списки статей — пагинация обязательна; при большом числе категорий кэширование списка категорий.
- Rich text: версионирование Payload rich text (lexical/slate) — убедиться в совместимости рендерера с форматом вывода API.

## 10. Definition of done (кратко)

- Архив, категория, статья работают по маршрутам; данные из Payload; только published.
- Rich text рендерится с заголовками, цитатами, ссылками, изображениями и alt.
- ToC автоматический; якоря работают.
- Просмотры через API, раз за сессию.
- Время чтения вычисляется; автор 8Blocks.
- SEO и noindex — по 18 и 11.
