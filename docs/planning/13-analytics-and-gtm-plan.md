# Analytics and GTM Plan

## 1. Purpose

Описать подключение Google Tag Manager и точки внедрения в приложении. Без реализации событий и тегов внутри GTM — только интеграция на стороне сайта.

## 2. Scope

- Подключение GTM по документации (script в head/body).
- Передача GTM ID через переменные окружения.
- Точки, где может потребоваться кастомный слой данных или триггеры (страницы, формы) — рекомендации для последующей настройки в интерфейсе GTM.

## 3. Inputs from requirements

- Подключена интеграция Google Tag Manager.
- Идентификатор GTM передаётся через переменные окружения.

## 4. Implementation

### 4.1 Переменная окружения

- Имя: `NEXT_PUBLIC_GTM_ID` (чтобы быть доступной на клиенте) или `GTM_ID` при внедрении только на сервере. Рекомендация: NEXT_PUBLIC_GTM_ID для клиентского скрипта GTM.
- Формат: строка вида `GTM-XXXXXX`.
- В .env.example указать: NEXT_PUBLIC_GTM_ID= (без значения).

### 4.2 Размещение кода GTM

- **Стандартная интеграция**: два фрагмента — один в <head> (как можно выше), второй после открытия <body>. Next.js: использовать next/script с strategy="afterInteractive" или вставить в layout.tsx через Script компонент или dangerouslySetInnerHTML в Head (по рекомендациям GTM для Next.js). Предпочтительно использовать официальный подход Next.js + GTM (документация Next.js и GTM).
- **Условие**: подключать скрипты только если NEXT_PUBLIC_GTM_ID задан; иначе не выводить тег, чтобы не было ошибок в консоли.

### 4.3 Контейнер и теги

- В интерфейсе GTM настраиваются теги (Google Analytics, прочие); на стороне сайта достаточно один раз подключить контейнер. Кастомные события (например отправка формы, клик CTA) при необходимости отправляются через dataLayer.push — опционально в этом плане; зафиксировать в документе возможность отправки событий с фронта для последующей настройки тегов в GTM.

## 5. Точки для аналитики (рекомендации)

- **Страницы**: смена страницы (SPA или полная перезагрузка) — по умолчанию GTM при полной загрузке страницы фиксирует просмотр; при Next.js client-side навигации может потребоваться dataLayer.push page_view или использование History Change trigger в GTM.
- **Формы**: успешная отправка формы обратной связи и подписки — при желании отправлять событие (например form_submit с параметром form_name) для конверсий.
- **CTA**: клики по кнопкам «Записаться на консультацию», «Подписаться» — при необходимости отдельные события через dataLayer.

Реализация событий — по решению продукта; в Phase 7 достаточно подключить GTM и убедиться, что контейнер загружается.

## 6. Dependencies

- Layout приложения (02, 01): место вставки скриптов — корневой layout.
- Env: NEXT_PUBLIC_GTM_ID в .env.example и в деплое.

## 7. Risks / open questions

- Consent (GDPR): ТЗ не требует cookie banner; при необходимости в будущем управление согласием и условная загрузка GTM — отдельная задача.
- Next.js App Router: корректное место для GTM (Script в layout, без блокировки гидрации).

## 8. Deliverables

- Подключение GTM в layout (или _document аналог): вывод script только при наличии NEXT_PUBLIC_GTM_ID.
- .env.example: NEXT_PUBLIC_GTM_ID=.
- Краткая заметка в документации: как подставить реальный ID и где при необходимости расширить dataLayer (формы, CTA).

## 9. Acceptance criteria

- При заданном NEXT_PUBLIC_GTM_ID контейнер GTM загружается на всех страницах.
- При отсутствии ID скрипты GTM не выводятся, ошибок нет.
- Документация обновлена.
